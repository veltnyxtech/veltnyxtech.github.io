What are API's

📝

An API, or Application Programming Interface, is one of the most fundamental concepts in modern software — and also one of the most commonly misunderstood by people just starting out. At its core, an API is simply a defined way for two pieces of software to communicate with each other, without either one needing to know the internal details of how the other actually works. It is a contract: a set of rules describing what requests are allowed, what data they should contain, and what response to expect back.

The word "interface" is doing the important work in the name. Just as a car's steering wheel and pedals form an interface that lets a driver control the vehicle without understanding its engine, an API forms an interface that lets a program interact with another program — a service, a library, an operating system — without needing to understand its internal implementation.

A Simple Analogy

The most common analogy, and a genuinely useful one, is a restaurant. A customer (the client) doesn't walk into the kitchen to cook their own meal. Instead, they look at a menu (the API's available endpoints), place an order with a waiter (send a request), and the kitchen (the server) prepares the food and the waiter brings it back (the response). The customer never needs to know how the kitchen is organized, what equipment it uses, or how the chef prepares the dish — they only need to know what's on the menu and how to order from it.

This is exactly how software APIs work. A weather app on a phone doesn't have its own network of meteorological sensors. Instead, it sends a request to a weather service's API — typically specifying a location — and receives back structured data describing the current conditions, which the app then displays in its own interface.

Types of APIs

  • Web APIs (REST, GraphQL): The most common type encountered in everyday software development, allowing applications to communicate over the internet using standard protocols like HTTP. REST APIs organize functionality around resources (like /users or /orders), while GraphQL APIs let a client specify exactly what data it needs in a single flexible query.
  • Library and framework APIs: The set of functions, classes, and methods a programming library exposes for other code to use. When a developer calls a function from a library like NumPy or React, they're using that library's API — a defined set of tools without needing to understand the library's internal source code.
  • Operating system APIs: Interfaces that let applications request services from the underlying operating system, such as reading a file, allocating memory, or accessing hardware like a camera or microphone.
  • Hardware APIs: Interfaces that let software communicate with physical hardware components, abstracting away the low-level details of how a specific chip or device actually operates.

How a Web API Request Actually Works

Since web APIs are the type most people encounter day-to-day — whether integrating a payment provider, pulling weather data, or connecting to a social media platform — it's worth walking through what actually happens during a typical request.

A client sends an HTTP request to a specific endpoint (a URL), specifying a method that describes the intended action: GET to retrieve data, POST to create something new, PUT or PATCH to update existing data, and DELETE to remove it. The request often includes headers (metadata like authentication tokens or content format) and, for POST or PUT requests, a body containing the actual data being sent, typically formatted as JSON.

The server receives this request, processes it — which might involve querying a database, running some business logic, or calling yet another API internally — and sends back a response. This response includes a status code indicating what happened (200 for success, 404 if the requested resource wasn't found, 500 if the server encountered an error, among many others), along with a response body containing the requested data or an explanation of what went wrong.

Why APIs Matter So Much

APIs are what make the modern software ecosystem possible. Without them, every application would need to build every piece of functionality itself from scratch — its own maps, its own payment processing, its own authentication system. Instead, a developer building a delivery app can use a mapping API for directions, a payments API for processing transactions, and an SMS API for sending order notifications, focusing their own effort on what makes their specific product unique rather than reinventing infrastructure that already exists and works well.

This same principle applies to internal software architecture as well. Large applications are often broken into smaller services that communicate through internal APIs, allowing different teams to build, update, and scale different parts of a system independently, as long as they all respect the agreed-upon API contract between them.

Common Points of Confusion

A few distinctions are worth clarifying for anyone new to the concept. An API is not the same thing as the internet, though web APIs do run over the same HTTP protocol used to load web pages — an API is the structured, machine-readable counterpart to a human browsing a website. An API is also not the same as an SDK (Software Development Kit); an SDK is a broader toolkit that often includes code, documentation, and helper libraries designed to make using a particular API easier, but the API itself is the underlying contract, while the SDK is one convenient way of interacting with it.

Understanding APIs at this foundational level — as a defined contract for communication between two independent pieces of software — makes it much easier to reason about everything built on top of that concept, from simple weather widgets to the complex, interconnected services that power most modern applications.
